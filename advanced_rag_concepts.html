Query Expansion

Bypassing Vocabulary Limitations

The Problem: Natural language questions are often short and mismatch the exact vocabulary inside business rules or glossary text files. For instance, a user asking for "top customer growth trend" might not get matches for rules containing the terms "net profit" or "cumulative billing increase".

The Solution: Query Expansion utilizes a language model to rewrite and broaden the user's prompt. By generating multiple variations, synonyms, or even hypothetical answers (HyDE - Hypothetical Document Embeddings), the retriever searches the vector DB using a combined pool of keywords.

Where it fits in our flow:
  • Runs inside the query analyzer module (nlp_analyzer.py) as a pre-retrieval layer.
  • Takes the initial query and generates 3 expanded query prompts.
  • Runs parallel embeddings queries on ChromaDB, returning a union of matching rules.
Python (Query Expansion)
def expand_query(user_query: str) -> list[str]:
    # Prompt template to expand intent
    prompt = f"""Rewrite this query into 3 distinct
variants focusing on different synonyms:
Question: {user_query}"""
    
    # LLM call returns variations
    variants = llm.generate(prompt) 
    
    # Always keep the original query
    return [user_query] + variants

# Usage in retrieval
expanded_queries = expand_query("growth trend")
all_docs = []
for q in expanded_queries:
    all_docs.extend(chroma_search(q))
# Remove duplicates afterward

Instead of asking the librarian for "Growth rules" (which finds nothing if the catalog uses "cumulative revenue increases"), you ask for "growth, sales trends, profit expansions, and billing spikes" all at once.

Reranking

Coarse Search to High-Precision Scoring

The Problem: Vector databases use Bi-Encoders (which embed queries and documents independently). This is extremely fast for searching millions of rows, but lacks the ability to capture deep cross-correlations. Often, the absolute best rule isn't in the top 3 spots.

The Solution: Use a two-stage retrieval pipeline. Stage 1 retrieves a larger pool of documents (e.g., top-K = 20) using standard cosine distance in ChromaDB. Stage 2 runs a heavier Cross-Encoder (Reranker) model to score the relationship between the query and each candidate chunk, placing the most critical rules at position 1.

Where it fits in our flow:
  • Positioned inside **rag_retriever.py** as a post-retrieval validation layer.
  • Reduces noise in the context builder, preventing context window pollution.
  • Significantly improves LLM comprehension by bringing relevant edge rules to the top.
Python (Reranking)
from sentence_transformers import CrossEncoder

# 1. Retrieve a wider set from ChromaDB
initial_hits = collection.query(
    query_embeddings=query, n_results=20
)['documents'][0]

# 2. Initialize Cross-Encoder (Reranker)
rerank_model = CrossEncoder(
    'cross-encoder/ms-marco-MiniLM-L-6-v2'
)

# 3. Predict similarity pairs
pairs = [(query, doc) for doc in initial_hits]
scores = rerank_model.predict(pairs)

# 4. Sort and return top 3
ranked_docs = sorted(
    zip(initial_hits, scores), 
    key=lambda x: x[1], reverse=True
)
top_3_rules = [doc for doc, score in ranked_docs[:3]]

Multi-Hop Retrieval

Solving Logical Chains & Joint Queries

The Problem: A single vector search can only match topics that are directly adjacent in semantic space. If a user asks "What is the profit margin of the supplier who ships the most orders?", a single search cannot fetch the answer. The supplier's identity must be resolved first before their specific rules can be retrieved.

The Solution: Multi-Hop retrieval runs queries in a structured sequence where the outputs of Hop 1 act as search keys for Hop 2. In SQL compiler terms, we find structural connections dynamically across tables and rules.

Where it fits in our flow:
  • Runs inside the context builder module (builder.py).
  • Hop 1: Query is analyzed to extract primary entity rules (e.g. Supplier orders).
  • Hop 2: Intermediate results (e.g. Supplier ID or name) are injected into the secondary lookup loop to extract relevant metric rules (e.g. margin calculation constraints).
Python (Multi-Hop Logic)
def multi_hop_retrieve(query: str):
    # Hop 1: Resolve main active entity
    hop1_context = lookup_engine.search_schema(query)
    primary_entity = hop1_context.get_top_entity() 
    # e.g., "Supplier"
    
    # Hop 2: Resolve sub-metrics for entity
    hop2_query = f"Margin rules for {primary_entity}"
    retrieved_rules = retriever.retrieve(hop2_query)
    
    # Compile unified context
    return build_context(hop1_context, retrieved_rules)

🔗 Detective Analogy

To solve a case, you can't just guess. First, you search for the suspect's address (Hop 1). Once you get the address, you look up the blueprints of that specific building (Hop 2).

Agentic RAG

Autonomous Reasoning and Self-Correction

The Problem: In standard RAG pipelines, data flows in one direction: Query ➔ Context ➔ SQL ➔ Print. If the SQL query fails to execute or a rule mismatch occurs, the pipeline fails silently or returns incorrect details.

The Solution: Wrap the retrieval and generation phases inside an autonomous reasoning loop (an LLM-based agent). The agent checks if the confidence score meets requirements. If it's too low, the agent uses tools (e.g. searching the glossary, running a SQL validator, asking the user a clarification question) to self-correct and refine until it guarantees accuracy.

Where it fits in our flow:
  • Replaces the simple static loop in **run_demo.py** with an iterative while-loop.
  • Connects the **ConfidenceScorer** and **ClarificationEngine** back into the retrieval loop.
  • Enables the system to test its own generated SQL queries in SQLite before outputting them.
Python (Agentic Reasoner)
def run_agentic_rag(query: str):
    context = []
    confidence = 0
    attempts = 0
    
    while confidence and attempts < 3:
        extracted = analyzer.analyze(query)
        rag_hits = retriever.retrieve(query, context)
        context = builder.build(extracted, rag_hits)
        
        confidence = scorer.calculate_score(context)
        if confidence >= 70:
            sql = generator.generate(context)
            # Self-Correction: Validate SQL
            if db.validate(sql):
                return sql
                
        # Agent decides to fetch glossary
        query = f"Explain glossary terms in query"
        attempts += 1

🤖 Chef Analogy

Instead of blindly following a recipe card and serving a bad dish, you taste the soup as you cook (evaluation), add seasoning if it is bland (re-retrieval), and check if the oven temperature is correct (self-correction).

Evaluation Metrics

Measuring Quality with RAGAS Framework

The Problem: How do you evaluate a RAG pipeline? Unit tests can check if database calls work, but they cannot tell you if the generated SQL is faithful to the text rules or if the retrieved chunks are full of useless noise.

The Solution: Implement standard metrics inspired by the RAGAS framework:

  • Faithfulness (0.0 - 1.0): Is the SQL logic strictly supported by the rules? Checks for hallucinations.
  • Answer Relevance (0.0 - 1.0): Does the query answer the user's intent?
  • Context Precision (0.0 - 1.0): Are retrieved chunks highly relevant to the query, or is there filler text?
  • Context Recall (0.0 - 1.0): Did we retrieve *all* rules required to generate the SQL query?
