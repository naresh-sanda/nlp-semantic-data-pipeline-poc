🧠 What is an Embedding? (An Analogy for Java Developers)

Think of a Vector Embedding as a GPS coordinate for meaning.

To define a physical location on Earth, you use 2 coordinates: [Latitude, Longitude]. Dallas, TX is located at [32.77, -96.79]. Fort Worth, TX is located at [32.75, -97.33]. Because their coordinate values are extremely similar, you can calculate the distance between them and know they are close. Tokyo, Japan is located at [35.67, 139.65], which is numerically far away.

Vector models do the exact same thing but for **semantic concepts** instead of geographical cities, using **384 coordinates** instead of 2. When the SentenceTransformer model parses the sentence "monthly revenue", it positions it at a specific location in a 384-dimensional space. The sentence "sales income net calculation" gets positioned very close to it, whereas "warehouse vendor supplier" is positioned far away.

🔬 Deep Dive: Chunking Mechanics

chunking.chunker.Chunker

Chunking is the process of breaking down large documents into smaller, distinct text blocks. It is essential because Machine Learning models have finite context token windows, and small, focused blocks yield cleaner embedding representations and avoid noise dilution in semantic searches.

Fixed Chunker

Splits text strictly by a character count (e.g. 200 characters). Fast and simple, but can cut words, sentences, or numbers in half, breaking semantic context.

Recursive Chunker

Splits text using hierarchical delimiters (double newlines first for paragraphs, then single newlines for sentences, then spaces) to keep semantic thoughts intact.

Metadata-Aware Chunker

Splits dynamically based on header boundaries (checks if a line starts with Term: or Rule ). Ensures related properties (like a rule description) remain grouped together.

💡 Java Equivalent
In Java, chunking is standardly done using splitting classes in LangChain4j (e.g., DocumentSplitters.recursive(...)) or Custom Lexers implementing boundary matchers.

🧠 Deep Dive: Vector Embeddings

phase1_ingestion.embeddings.embedder.Embedder

An embedding is a numerical representation of unstructured text. It maps words and sentences into a high-dimensional vector space. The key feature is that text blocks with similar semantic meanings will reside close to each other in this coordinate space, regardless of the exact vocabulary used.

  • Underlying Model: all-MiniLM-L6-v2 is a highly efficient transformer model trained to convert text inputs of up to 256 tokens into coordinate vectors.
  • Vector Dimensionality: Produces vectors containing exactly 384 floating-point coordinates.
  • Vector Database Storage: ChromaDB acts as the local storage indexing engine (analogous to a vector-optimized Lucene index or Elasticsearch index), saving vectors to disk under vector_store/.
💡 Java Equivalent
In Java, this is mapped to an EmbeddingModel (e.g. AllMiniLmL6V2EmbeddingModel from LangChain4j) and stored in vector repositories like Chroma, Milvus, or Pgvector.

📐 Deep Dive: Similarity Math & Hybrid Re-ranking

phase2_retrieval.retrieval.rag_retriever.RAGRetriever

Vector search calculates proximity by comparing query vectors against indexed vectors. The pipeline uses two core algorithms to filter and rank rules:

1. Distance to Cosine Similarity Conversion:
similarity = max(0, 1 - (L2_distance / 2))
ChromaDB defaults to Euclidean distance (squared L2 distance). Because SentenceTransformer vectors are unit-normalized (length = 1), their squared L2 distance spans from 0 to 4. We map this back to standard Cosine Similarity (0 to 1) for scoring.
2. Hybrid Scoring Equation (Combining Vector Search & Keyword relevance):
final_hybrid_score = (similarity * 0.7) + (bm25_score * 0.3)
We implement a hybrid ranking mechanism: Vector search models semantic meaning (70% weight) while keyword matching (BM25 simulation, 30% weight) ensures exact keywords are not lost, producing much cleaner retrieval under short query inputs.
💡 Java Equivalent
Calculations like Cosine Similarity or dot products are built directly into LangChain4j embedding store matching algorithms or executed at database level (e.g. pgvector <=> operator).

Stage 0: Initialization Flow

phase1_ingestion.loaders.metadata_loader | phase1_ingestion.embeddings.embedder

When you run the ingestion pipeline (python run_phase1_ingestion.py), the script initializes the workspace storage layers:

  1. SQLite setup (setup_database): Reads metadata CSV/JSON files from metadata/ directory (like semantic_entities.csv) and loads them into local SQLite database db/metadata.db using Pandas (similar to mapping files directly into H2 tables using JDBC/Hibernate).
  2. ChromaDB setup (initialize_knowledge_base): Reads unstructured rules (semantic_business_rules.txt) and glossary (semantic_glossary.txt). The Chunker splits files into discrete semantic entries (like separate glossary terms or business rules). The Embedder loads the SentenceTransformer('all-MiniLM-L6-v2') model, converts these text blocks into 384-dimension vector embeddings, and stores them in ChromaDB (like a search index).

Stage 1: NLP Question Analysis

phase2_retrieval.analyzer.nlp_analyzer.NLPAnalyzer

Analyzes raw string input to parse tokens into logical categories: metrics, entities, time_grain, and filters.

Stage 2: Metadata Lookup

phase2_retrieval.metadata_lookup.lookup_engine.LookupEngine

Queries the local SQLite database to resolve extracted keywords into real database mappings:

  • Entities: Finds matching rows in semantic_entities by comparing synonyms using a SQL LIKE query (e.g. WHERE synonyms LIKE '%supplier%').
  • Metrics: Finds corresponding rows in semantic_metrics (e.g. resolving keyword "margin" to metric ID m3 named Margin with formula calculation).
  • Relationships: Queries semantic_relationships to identify whether the resolved entities have any documented relationships (e.g. Customer has a one-to-many relationship with Order).

Stage 3: Semantic RAG Retrieval

phase2_retrieval.retrieval.rag_retriever.RAGRetriever

Vector databases enable retrieving context based on meaning rather than exact keyword matches:

  1. Converts the user's natural language query into a vector representation (384-dimensional array) using the Sentence Transformer model.
  2. Sends this query vector to ChromaDB to retrieve the top 3 closest document chunks based on Cosine Similarity.
  3. Filters out chunks that fall below a similarity threshold (e.g., 0.3) and ranks them using a hybrid score combining vector search distance and simulated keyword relevance (BM25).

Stage 5: SQL Generator & dynamic Joins

phase2_retrieval.sql_generator.generator.SQLGenerator

Translates the mapped entities, metrics, grain, and filters into a valid SQL query.

🔍 Dynamic Join Resolution Logic
If multiple tables are identified in the query, the generator automatically determines join keys:
  • customers c & orders o → c.customer_id = o.customer_id
  • regions r & orders o → r.region_id = o.region_id
  • products p & orders o → p.product_id = o.product_id
  • suppliers s & orders o → bridge automatically using products p and add join predicates: s.supplier_id = p.supplier_id AND p.product_id = o.product_id
