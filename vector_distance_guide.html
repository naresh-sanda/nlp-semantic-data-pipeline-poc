🎯 The Big Picture

When a user types "Show monthly revenue by customer", your pipeline needs to find the most relevant business rules and glossary entries from ChromaDB. But computers don't understand English — they understand numbers.

The embedding model (all-MiniLM-L6-v2) converts every piece of text into a list of 384 numbers called a vector. The question then becomes:

"Which stored vectors are closest to the query vector?"

"Closest" is where the distance metric comes in — it's the ruler you use to measure how similar two vectors are. There are three common rulers, and each measures "closeness" differently.

📏 The Three Rulers

🔵

Squared L2 (Euclidean)

"Straight-Line Distance"
  • Measures physical distance between two points
  • Lower value = more similar
  • Range: 0 (identical) → ∞ (very different)
  • Cares about vector magnitude (length)
  • ChromaDB's default metric
🟢

Cosine Similarity

"Direction, Not Distance"
  • Measures the angle between two directions
  • Lower distance = more similar
  • Range: 0 (identical) → 2 (opposite)
  • Ignores magnitude (text length)
  • Industry standard for RAG pipelines
🔴

Inner Product (Dot Product)

"Direction × Confidence"
  • Measures direction AND magnitude combined
  • Higher value = more similar
  • Range: -∞ → +∞
  • Rewards both alignment and importance
  • Only use when model docs say to

🏠 Library Analogy

🔵 L2 — Straight-Line Distance

You're standing in a library. You measure the physical walking distance from you to each book on the shelves. The nearest book wins. But a book on a high shelf (long document = big vector) will seem "farther away" even if it's about the exact same topic.

🟢 Cosine — Direction Only

Instead of measuring walking distance, you stand at the center and point at each book. You measure the angle between your arm pointing at your topic and your arm pointing at each book. Smaller angle = more similar. It doesn't matter if the book is on a high shelf or low shelf — only the direction matters.

🔴 Inner Product — Direction × Popularity

Same as cosine (direction matters), but books that the library has more copies of (higher magnitude) get a bonus. Popular relevant books rank higher than obscure relevant books.

🔢 See It In Real Numbers

Let's use tiny 3-number vectors to make this concrete. In reality your model uses 384 numbers, but the math works the same way.

Setup: One Query, Two Candidate Chunks
Query vector:
[0.8, 0.3, 0.1] ← "revenue by customer"
Book A vector:
[0.7, 0.4, 0.2] ← "Revenue is net amount after discounts"
Book C vector:
[0.1, 0.1, 0.9] ← "Warehouse storage location"

🔵 L2 Distance Calculation

L2 MATH
// For each pair of numbers: subtract, then square

// Query → Book A (Revenue rule)
L2 = (0.8-0.7)² + (0.3-0.4)² + (0.1-0.2)²
   = 0.01 + 0.01 + 0.01 = 0.03  ← SMALL = similar! ✅

// Query → Book C (Warehouse info)
L2 = (0.8-0.1)² + (0.3-0.1)² + (0.1-0.9)²
   = 0.49 + 0.04 + 0.64 = 1.17  ← BIG = not similar ❌

🟢 Why Cosine Handles Length Differences Better

Now imagine Book A is a long document — its vector gets scaled up:

THE LENGTH PROBLEM
// Same meaning, but the document is 3x longer
Query (short):      [0.8, 0.3, 0.1]
Book A (long doc):  [2.1, 1.2, 0.6]  ← same direction, bigger numbers

// L2 says: FAR APART (bad!)
L2 = (0.8-2.1)² + (0.3-1.2)² + (0.1-0.6)²
   = 1.69 + 0.81 + 0.25 = 2.75  ← Seems far! ❌

// Cosine says: SAME DIRECTION (correct!)
// Both vectors point the same way, just different lengths
Cosine similarity = 0.98  ← Nearly identical meaning! ✅
Visual: Why Cosine Ignores Length
Query (short) → Book A (long)
Same direction, different length
Cosine: 0.98 ✅
L2: 2.75 (misleading!)
Query → Book C (Warehouse)
Different direction entirely
Cosine: 0.26 ❌
L2: 1.17 ❌

🔍 Which One Fits Our Pipeline?

Our Data Characteristics

WhatExampleLength
User query (short) "Show monthly revenue by customer" ~6 words
Business rule chunk "Rule 1: Revenue Calculation — Revenue should always be calculated using the net amount after discounts..." ~20-30 words
Glossary chunk "Term: Churn — The rate at which customers stop doing business with an entity." ~15-20 words

The query is short (~6 words), while matching chunks are 3-5x longer. With L2, the vectors would have different magnitudes, and L2 would think they're farther apart than they actually are. Cosine ignores this size difference.

Use Cosine Similarity for Our Pipeline

Short queries vs. longer chunks Cosine ignores magnitude — a 6-word query matches a 30-word rule perfectly
Our model (all-MiniLM-L6-v2) Produces normalized vectors — cosine and L2 give equivalent rankings, but cosine scores are more readable
Mixed chunk sizes Our metadata-aware chunker produces chunks of varying length
Industry standard Almost all RAG pipelines use cosine similarity for text retrieval
About Normalized Vectors: all-MiniLM-L6-v2 produces vectors where all magnitudes are exactly 1.0. When vectors are normalized, L2 and cosine give the same ranking. The practical benefit of cosine is that similarity scores are human-readable: 0.95 clearly means "95% similar", while an L2 distance of 0.31 doesn't tell you much without context.

📁 Where This Lives In Our Code

Step 1: Collection is created in the Embedder

File: embedder.py — line 19

CURRENT CODE
# Line 19 — THIS is where the distance metric is chosen
self.collection = self.client.get_or_create_collection(
    name="semantic_knowledge"
)
# ⚠️ No distance metric specified → ChromaDB defaults to L2!

To use cosine instead:

RECOMMENDED CHANGE
self.collection = self.client.get_or_create_collection(
    name="semantic_knowledge",
    metadata={"hnsw:space": "cosine"}    # ← sets the distance metric
)

Step 2: Query returns distances in the RAG Retriever

File: rag_retriever.py — lines 34-44

CURRENT CODE
# Lines 34-37 — ChromaDB returns results with distances
results = self.collection.query(
    query_embeddings=query_embedding,
    n_results=top_k + 2
)

# Lines 41-44 — Converting distance to similarity
for doc, distance in zip(results['documents'][0], results['distances'][0]):
        # Currently: assumes L2 and approximates cosine
        similarity = max(0, 1 - (distance / 2))
        # With cosine space: distance = (1 - cosine_similarity)
        similarity = max(0, 1 - distance)  # Clean 0.0 → 1.0 range

What ChromaDB Returns

🔵 L2 (current) [0.35, 0.78, 1.45] Lower = closer. Range: 0 → ∞. Not intuitive.
🟢 Cosine (recommended) [0.05, 0.22, 0.67] Lower = closer. 1 - 0.05 = 0.95 similarity. Very intuitive!
Important: If you switch the distance metric, you must delete the existing vector_store/ folder and re-run Phase 1 ingestion. ChromaDB stores the metric choice at collection creation time and cannot change it afterwards.

🗺️ Quick Decision Guide

Use this flowchart whenever you need to choose a distance metric for any vector search project:

Do your chunks vary significantly in length?
├── YES → Use Cosine ✅ (← our case!)
└── NO (all chunks ~same size)
    ├── Does your model documentation say to use Inner Product?
    │   ├── YES → Use Inner Product
    │   └── NO → Use L2 (simplest, fastest)
    └── Not sure? → Use Cosine (safest default)
