STEP 1

Metadata Database Loading

Primary Function setup_database(db_path, metadata_dir)
Input Assets CSV files (tables.csv, columns.csv, joins.csv)
Output Asset SQLite DB file (metadata.db)

Parses schema configuration files (CSV) defining the structure of the data warehouse. This metadata registers database tables, columns, data types, and primary-to-foreign key relationships. The loader parses the CSV structures and initializes standard SQLite relational schemas to power high-speed schema mapping at runtime.

metadata_loader.py (Core Logic Snippet) 
import sqlite3, csv
def setup_database(db_path, metadata_dir):
    conn = sqlite3.connect(db_path)
    cursor = conn.cursor()
    # Create tables representing relational schema
    cursor.execute("""CREATE TABLE IF NOT EXISTS tables (
        table_name TEXT PRIMARY KEY, description TEXT, type TEXT
    )""")
    # ... (create column & joins tables) ...
    
    # Populate from metadata folder CSVs
    with open(f"{metadata_dir}/tables.csv", 'r') as f:
        reader = csv.reader(f)
        next(reader) # skip header
        cursor.executemany("INSERT OR REPLACE INTO tables VALUES (?,?,?)", reader)
    conn.commit()
    conn.close()
☕ Java Implementation Approach

In Java, this ingestion is standard schema initialization. Typically managed via:

  • Spring Boot Schema Loader: Run SQL statements utilizing schema.sql and data.sql dynamically at runtime.
  • CSV Parser: Use Apache Commons CSV or OpenCSV to parse structural configuration files.
  • Data Insertion: Run batch updates via JdbcTemplate.batchUpdate(...) for transaction performance.
STEP 2

Document Chunking

Primary Class Chunker
Input Assets TXT Rules (semantic_business_rules.txt, glossary.txt)
Output Asset List of Text Strings (Chunks)

Splits semantic documents into smaller, coherent text chunks to ensure efficient vector index lookups. The Chunker supports different strategies, including fixed character splitting, recursive paragraph-splitters, and a custom metadata-aware splitter designed specifically to group custom business rules (splitting text sections beginning with "Term:" or "Rule:").

chunker.py (Metadata-Aware Logic) 
class Chunker:
    def _metadata_aware_chunking(self, text, metadata=None):
        lines = text.split("\n")
        chunks, current_chunk = [], []
        for line in lines:
            # Split logically when a rule or glossary term is declared
            if line.startswith("Term:") or line.startswith("Rule:"):
                if current_chunk:
                    chunks.append("\n".join(current_chunk))
                current_chunk = [line]
            else:
                current_chunk.append(line)
        if current_chunk:
            chunks.append("\n".join(current_chunk))
        return [c for c in chunks if c.strip()]
☕ LangChain4j Equivalents

LangChain4j provides robust built-in objects to handle document parsing and splitting:

  • Document Splitter: Map this operation directly to LangChain4j's DocumentSplitters. E.g., DocumentSplitters.recursive(300, 30) split by characters/paragraphs.
  • Custom Parser: For metadata-aware boundary matching, implement the DocumentSplitter interface and code a custom loop matching line.startsWith("Term:"), returning a list of TextSegment wrappers.
STEP 3

Vector Store Indexing

Primary Class Embedder
ML Model SentenceTransformer('all-MiniLM-L6-v2')
Target DB ChromaDB Persistent Client

Initializes the local Machine Learning embedding model and registers chunks in the vector database. It converts character arrays (text chunks) into 384-dimensional floating point vectors representing semantic meaning. The vector coordinates are upserted into an offline ChromaDB index (`vector_store/`), making them fully searchable by distance metrics (e.g. Cosine Similarity) at runtime.

embedder.py (Embedding & Upsert) 
import chromadb
from sentence_transformers import SentenceTransformer

class Embedder:
    def __init__(self, db_dir):
        self.client = chromadb.PersistentClient(path=db_dir)
        self.collection = self.client.get_or_create_collection("semantic_knowledge")
        self.model = SentenceTransformer('all-MiniLM-L6-v2')

    def add_chunks(self, chunks):
        # Generate embedding vector lists offline
        embeddings = self.model.encode(chunks).tolist()
        ids = [f"chunk_{i}" for i in range(len(chunks))]
        # Insert vector coordinate records into ChromaDB
        self.collection.upsert(
            embeddings=embeddings, documents=chunks, ids=ids
        )
☕ Java Library Mapping

To run embeddings and vector storage natively on JVM:

  • Local Embedding Model: Import dev.langchain4j.model.embedding.AllMiniLmL6V2EmbeddingModel. This loads the exact same model running inside an in-memory Java **ONNX Runtime** instance.
  • Vector Database Storage: Configure an EmbeddingStore client wrapper pointing to your database, such as ChromaEmbeddingStore, MilvusEmbeddingStore, or PgVectorEmbeddingStore.

STEP 1

NLP Intent Analysis

Primary Method NLPAnalyzer.analyze(query)
Input User Natural Language Query (string)
Output JSON Object (extracted entities, intent, keywords)

Evaluates the incoming query string using regex-based pattern matching (intent parser) to identify query targets (e.g. "revenue", "margin"), filters, aggregation fields (e.g. "monthly"), and candidate table/column references. This allows the system to build structured search tokens for both the database metadata and the vector collection.

nlp_analyzer.py (Intent Parsing) 
import re
class NLPAnalyzer:
    def analyze(self, query):
        q = query.lower()
        extracted = {"intent": "select", "entities": [], "aggregations": []}
        
        # Detect aggregation terms
        if "monthly" in q or "month" in q:
            extracted["aggregations"].append("monthly")
            
        # Extract keyword candidates using boundary patterns
        words = re.findall(r'\b[a-zA-Z]{3,}\b', q)
        extracted["entities"] = [w for w in words if w not in ["show", "find", "list"]]
        return extracted
☕ Java Parsing Options

To parse query intent and identify entity keywords on the JVM:

  • Java Regular Expressions: Direct match using standard packages (java.util.regex.Pattern and Matcher).
  • NLP Engines: For production-level entity tagging without complex regex, use **Apache OpenNLP** (specifically the TokenizerME and NameFinderME classes) to identify keywords and structural concepts.
STEP 2

SQLite Schema Lookup

Primary Method LookupEngine.lookup(extracted_data)
Input Extracted Entity Keywords (dict)
Output Database schemas, matching tables, primary join keys

Queries the SQLite metadata database (`db/metadata.db`) using the parsed entities. It searches the tables, columns, and joins indexes to identify matching table schemas and the primary/foreign key connections required to dynamically construct join statements (e.g. `customer_id` join paths).

lookup_engine.py (SQLite SQL Queries) 
import sqlite3
class LookupEngine:
    def __init__(self, db_path):
        self.db_path = db_path

    def lookup(self, extracted_data):
        conn = sqlite3.connect(self.db_path)
        cursor = conn.cursor()
        entities = extracted_data.get("entities", [])
        matches = []
        for ent in entities:
            # Query SQLite database for tables or columns containing this concept
            cursor.execute("""SELECT table_name, description FROM tables 
                             WHERE table_name LIKE ? OR description LIKE ?""", 
                           (f"%{ent}%", f"%{ent}%"))
            matches.extend(cursor.fetchall())
        conn.close()
        return matches
☕ Java Database Interaction

Querying relational metadata in Java is standard database access:

  • Spring JDBC / MyBatis: Execute matching queries using NamedParameterJdbcTemplate or JPA/Hibernate queries mapping to a relational entity class (e.g. MetadataTable).
  • In-Memory Database: If metadata updates are rare, load them into a local cache or a nested **H2 Database** instance to achieve sub-millisecond query responses.

Semantic RAG Search

Primary Method RAGRetriever.retrieve(query)

STEP 4

LLM Prompt Context Assembly

Primary Method ContextBuilder.build_context(schemas, rules)
Input Arrays Schema list + retrieved RAG rule list
Output Structure Context dictionary (JSON/text format)

Consolidates the output of Step 2 (exact relational database schemas) and Step 3 (semantic business rules retrieved from ChromaDB). The ContextBuilder groups, formats, and structures these elements into a prompt payload, providing the downstream LLM with all context needed to generate syntactically correct SQL.

builder.py (Context Formatting) 
class ContextBuilder:
    def build_context(self, lookup_results, rag_results):
        schema_text = "\n".join([f"Table: {r[0]}, Description: {r[1]}" for r in lookup_results])
        rules_text = "\n".join([f"- {item['chunk']}" for item in rag_results])
        
        return {
            "schema_context": schema_text,
            "business_rules": rules_text,
            "formatted_prompt": f"SCHEMAS:\n{schema_text}\n\nRULES:\n{rules_text}"
        }
☕ Java Prompt Construction

Templating and formatting context inside JVM:

  • LangChain4j PromptTemplate: Define reusable prompts using structured files or String declarations. E.g., PromptTemplate.from("SCHEMAS:\n{{schemas}}\n\nRULES:\n{{rules}}").
  • Binding Data: Map values dynamically by passing a Java Map<String, Object> structure directly to the template.apply(variables) execution helper.
STEP 5

SQL Query Generation

Primary Method SQLGenerator.generate(context, parsed, query)
Logic Model Rule-based query generation & LLM simulation
Output SQL Standard SQL Query String

Processes the compiled prompt context to generate the executable SQL query. For the local offline POC environment, the SQLGenerator uses a deterministic template-matching strategy that maps the parsed intent, detected entities, and join columns into standard ANSI SQL.

generator.py (SQL Blueprint Mapping) 
class SQLGenerator:
    def generate(self, context, extracted_data, query):
        # Simple matching logic representing the dynamic generation step
        intent = extracted_data.get("intent", "select")
        entities = extracted_data.get("entities", [])
        
        if "revenue" in entities:
            sql = "SELECT SUM(s.quantity * s.unit_price) AS revenue FROM sales s"
            # Add join paths dynamically if customers metadata matches
            if "customer" in entities:
                sql += " JOIN customers c ON s.customer_id = c.customer_id"
            # Add grouping clauses based on aggregations
            if "monthly" in extracted_data["aggregations"]:
                sql += " GROUP BY DATE_TRUNC('month', s.sale_date)"
            return sql
        return "SELECT * FROM sales LIMIT 10"

STEP 6

Confidence Scoring & Clarification Gate

Primary Classes ConfidenceScorer & ClarificationEngine
Threshold Limit score >= 0.70 (Pass Gate)
Response Action Run Query or Prompt User for Clarification

Acts as the quality gatekeeper of the pipeline. The ConfidenceScorer calculates a numerical score (0.0 to 1.0) based on: matching schema keys, table matches, and vector distance values. If the score falls below a threshold (e.g. 0.7), the ClarificationEngine intercepts the flow to prompt the user for clarification (simulating conversational disambiguation).

scorer.py (Evaluation Framework) 
class ConfidenceScorer:
    def calculate_score(self, context, extracted_data, rag_results):
        score = 0.5  # baseline
        # Boost if schema lookup successfully matched tables
        if context.get("schema_context"):
            score += 0.3
        # Boost based on high semantic match coefficients
        if rag_results and rag_results[0]["final_hybrid_score"] > 0.6:
            score += 0.2
        return min(1.0, score)

class ClarificationEngine:
    def check_needs_clarification(self, score, extracted, context):

☕ Java Interceptor Pattern

Implementing the scoring validator and fallback routing in Java:

  • Interceptor / Validator: Design a Spring Boot component implementing an Adapter/Interceptor pattern. It intercepts the pipeline return object before execution.
  • Dynamic Routing: If the model's confidence field is insufficient, throw a custom checked exception (e.g. LowConfidenceException). This is caught by your REST controller to return a `202 Accepted` status with an array of clarifying UI checkboxes.
