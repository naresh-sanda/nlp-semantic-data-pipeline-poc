🔬 Phase 1 Ingestion Details & Libraries

Python Implementation

python standard re regex SentenceTransformer (all-MiniLM-L6-v2) chromadb

The ingestion workflow reads raw text files from disk. First, a local regex-based chunker splits the text using header boundary tokens. Then, the SentenceTransformer library loads the local all-MiniLM-L6-v2 model to convert chunks into coordinates. Finally, the embedding vectors and original text documents are saved as a collection inside ChromaDB.

Java Equivalent

LangChain4j Core DocumentSplitters AllMiniLmL6V2EmbeddingModel ChromaEmbeddingStore

In Java, this ingestion flow maps cleanly to the LangChain4j library. Documents are parsed into Document wrappers, chunked using DocumentSplitters (such as a recursive splitter), embedded using an EmbeddingModel client (calling locally or an API), and stored in an instance of EmbeddingStore (like Chroma, Milvus, or Pgvector).

💾 Relational DB vs. Vector DB: Storage Comparison (Phase 1)

To store text in a regular database, you run an SQL INSERT statement. In a Vector Database, you insert the text document, its ID, optional metadata fields, and its generated embedding vector coordinates. The vector coordinates act as a semantic "index" that maps where the text fits in coordinate space.

Concept Relational DB (SQL DB) Vector DB (ChromaDB / Milvus)
Logical Container Table (strict rows and columns) Collection (unstructured documents and indexes)
Stored Records Rows (standard columns like ID, content, type) Documents (ID + Text Document + Metadata + Embeddings Vector)
Connection & Hosting TCP Server (e.g., Postgres / MySQL server running on port 5432) or In-Process File (SQLite file sqlite3.connect("db.db")).

In Java, you configure a DataSource connection pool (like HikariCP) using a JDBC URL: 
jdbc:postgresql://localhost:5432/db
 Supports two main deployment structures:

1. Embedded Mode: Runs directly inside the application process (stores index files to disk).

2. Server Mode: Connects via HTTP REST APIs to a remote vector database instance.

import dev.langchain4j.data.document.Document;
import dev.langchain4j.data.document.splitter.DocumentSplitters;
import dev.langchain4j.model.embedding.AllMiniLmL6V2EmbeddingModel;
import dev.langchain4j.store.embedding.chroma.ChromaEmbeddingStore;

// 1. Initialize Store and Model
EmbeddingStore<TextSegment> store = ChromaEmbeddingStore.builder()
    .baseUrl("http://localhost:8000")
    .collectionName("knowledge")
    .build();
EmbeddingModel model = new AllMiniLmL6V2EmbeddingModel();

// 2. Chunking & Encoding
Document document = Document.from("Rule 1: Revenue... \\nRule 2: Profit...");
DocumentSplitter splitter = DocumentSplitters.recursive(200, 20);
List<TextSegment> segments = splitter.split(document);

// 3. Storage
for (TextSegment segment : segments) {
    Embedding embedding = model.embed(segment).content();
    store.add(embedding, segment);
}

💡 Phase 1 Ingestion: Developer Tips & Gotchas

1. Chunks Tuning is an Art

Do not guess your chunk sizes. If your business rules are long, a chunk size of 200 characters might cut sentences in half. A chunk size of 1000 characters with an overlap of 100 is standard for text paragraphs. If chunking rules, split by logical dividers (like double newlines or bullet headers) instead of characters.

2. Thread-Safety in Vector Databases

ChromaDB running in Embedded Persistent Mode is backed by SQLite and writes to disk. In a multi-threaded web server (like FastAPI in Python or Spring Boot in Java), concurrent inserts can lead to database locked exceptions. Ensure your database client instances are managed as singletons.

3. Java Library Gotchas (LangChain4j)

In LangChain4j, running local models like AllMiniLmL6V2EmbeddingModel uses native ONNX runtime bindings. On Windows/Linux servers, make sure you have the correct OS permissions and native compilation headers if the library attempts to load system-specific libraries.

4. Embedding Re-indexing Cost

Generating embeddings is CPU/GPU intensive. Do not clear and re-insert the collection every time your application boots. Check if the database files already exist, or keep a metadata hash of files to only update modified lines.

🔬 Phase 2 Runtime Details & LLM Orchestration

Python Implementation

sentence_transformers chromadb collection.query() L2-to-Cosine distance translation openai / google-generativeai API

When the user asks a question, the pipeline embeds the query using all-MiniLM-L6-v2. It requests coordinates matching against ChromaDB. The Euclidean distance is mapped to a cosine similarity metric: similarity = 1 - (distance / 2). The top rules exceeding the threshold are combined with schema lookups. An LLM Prompt Template is sent to OpenAI or Gemini, which responds with clean SQL. If the LLM throws a connection error or keys are missing, the pipeline automatically redirects execution to the local deterministic template system.

Java Equivalent

LangChain4j AiServices ChatLanguageModel PromptTemplate / SystemMessage Query Router / Fallback Handler

In Java, runtime orchestration is handled via LangChain4j AiServices. The user query is sent to a service interface. Under the hood, LangChain4j queries the EmbeddingStore, formats retrieved matching rules into a SystemMessage template along with DB schemas, and invokes a ChatLanguageModel (e.g. OpenAI/Gemini/Ollama). To implement the fallback, developers write a Java Try-Catch handler wrapping the LLM service method that returns a predefined JDBC/MyBatis SQL template if the remote API fails.

🔍 Relational DB vs. Vector DB: Retrieval Comparison (Phase 2)

To retrieve rules in a relational database, you must search by exact keywords (e.g., using LIKE '%margin%'). If the user types a synonym like "profit analysis", the relational search yields zero results. A Vector DB converts the user query into a vector and finds the closest coordinate points, retrieving the correct rule even with zero keyword overlap.

Concept Relational DB (SQL Query) Vector DB (Similarity Query)
Search Metric Keyword Matching / SQL LIKE query (exact string patterns) Coordinate Proximity / Similarity Score (semantic meaning)
Retrieval Code

FROM rules_table

🛡️ The Gatekeeper: Confidence Scorer & Clarification Engine (Phase 2 Only)

To protect the data warehouse from executing hallucinated or inaccurate queries, a Confidence Scorer acts as a gatekeeper. It evaluates the compiled database context and vector search matches before any SQL is generated.

🔄 Phase 2 Execution Order (Context-First, Scoring-Second, LLM-Third)
User Question │ ▼ Retrieve Rules (Vector DB) + Metadata (SQLite) │ ▼ [Build Context Object] │ ▼ Calculate Confidence Score (Is it >= 70?) ├── Yes ──► Pass Context to LLM ──► Generate SQL Query │ └── No ──► Trigger Clarification Engine ──► Ask User to Clarify (Halt SQL)

Why is it ONLY in Phase 2?

Phase 1 (Ingestion & Storage) is a static, one-time pipeline that chunks documents and stores them in ChromaDB. There are no queries to evaluate or intent matches to perform, meaning confidence scoring is irrelevant here.

Phase 2 (Retrieval & Generation) is a dynamic runtime process. A user query can be ambiguous, incomplete, or poorly mapped. The Confidence Scorer evaluates the completeness of the mapped schema metadata and retrieved business rules at runtime, preventing the LLM from making guesses.

The Clarification Threshold (< 70)

If the evaluated score falls below 70, the pipeline execution stops and triggers the Clarification Engine. Rather than executing a low-confidence SQL query, the engine identifies the missing component (dimensions or metrics) and asks the user:

  • Missing Metrics: "Which specific metric are you looking for? (e.g., Gross Revenue, Net Revenue, Profit?)"
  • Missing Dimensions/Entities: "By which dimension would you like to see this? (e.g., by Customer, by Region?)"

Dual Scoring Methodologies

LLM Scorer (Primary) Rule-Based Math (Fallback)

1. LLM Assessor (Primary Mode)

Constructs a prompt sending the user question, the matched schema entities/metrics/relationships, and the text of retrieved rules. It asks the LLM to inspect schema coverage, check for unresolved relationships, and output a structured JSON containing a score (0-100) and reasoning.

2. Deterministic Fallback Scorer (Local Rule-Based)

If LLM keys are missing or connections time out, the scorer falls back to a deterministic point rubric:

Factor Evaluated Max Points Rule Metric
Metadata Match 30 pts 15 pts if entities matched, 15 pts if metrics matched.
Relationship Match 20 pts 20 pts if joins are resolved/unneeded. 5 pts if relationships are missing for multiple entities.
Similarity & Rules 35 pts Average vector cosine similarity score of retrieved rules scaled out of 35.
Question Understanding 15 pts 15 pts if metrics or entities are parsed in NLP intent.
Total Possible 100 pts Gate threshold >= 70 to execute SQL generation.

💡 Phase 2 Retrieval: Developer Tips & Gotchas

1. Semantic Search Similarity Scaling

Vector similarity distance metrics vary. ChromaDB defaults to L2 Squared distance (where 0.0 is an exact match and larger values mean different coordinates). To convert it to a Cosine Similarity score from 0.0 to 1.0, you must use scaling formulas: similarity = 1.0 - (l2_distance / 2.0).

2. Handle Context Overflow (Token Budgets)

When retrieving multiple documents (using Top-K match), you might exceed the LLM's context window. Always track your token budgets and truncate matches before passing the prompt payload. A common error is passing a massive schema which leaves 0 tokens for the LLM's response.

3. API Failures & Rate Limits

LLM API calls are highly unstable. In Java, wrap your ChatLanguageModel calls inside retry interceptors or resilience frameworks like Resiliance4j. In Python, use decorators like @retry. The local template fallback mechanism is your final safety net.

4. Match Thresholding

Vector search will always return results, even if the user query is completely unrelated. For example, asking "what is the weather?" will still return the closest business rule in your database. Always set a minimum similarity threshold (e.g. 0.35) below which you ignore the matching rule context.
