⚙️ Advanced RAG & Vector Concepts

📚 1. Chunk Overlap Strategy
During Phase 1 (Ingestion), large documents are split into chunks. If you split text strictly on character boundaries, you run the risk of cutting a critical word, sentence, or calculation formula in half.

Why we use Overlap: By keeping a sliding window (e.g. chunk size 200 characters, overlap 20), the end of Chunk A matches the beginning of Chunk B. This guarantees that context near boundary splits is not lost or fragmented, preserving mathematical formulas and vocabulary.
# No Overlap (Clipped context) Chunk 1: "...Profit Margin is calculated as (Revenue -" Chunk 2: "Cost) / Revenue. It should always..." # With Overlap (Context preserved) Chunk 1: "...calculated as (Revenue - Cost)" Chunk 2: "calculated as (Revenue - Cost) / Revenue..."
📐 2. Vector Indexing: HNSW vs. Flat (Full Scan)
When querying vectors in ChromaDB or Milvus, calculating distances against millions of records is computationally expensive.

Flat Index (No Graph): Performs a linear search (full scan), calculating distance coordinates against every vector in the DB. Slow on large databases.

HNSW (Hierarchical Navigable Small World): Builds a multi-layered graph where the top layers contain sparse networks for quick routing, and lower layers contain dense, localized coordinate networks. This acts like a **Relational Clustered Index / B-Tree**, allowing the database to search logarithmic paths instead of a full scan.
[HNSW Hierarchical Graph Structure] Layer 2 (Express): (Node A) ------------ (Node G) | | Layer 1 (Fast): (Node A) --- (Node D) --- (Node G) | | | Layer 0 (All docs): (A)-(B)-(C) -- (D)-(E) -- (F)-(G)
⚡ 3. Pre-Filtering vs. Post-Filtering
If your database rules belong to specific workspaces or users, you want to restrict search boundaries.

Post-Filtering: First fetches the top 10 closest vectors globally, and then discards results that do not belong to the user's workspace. Caution: If the top 10 results belong to other workspaces, the user receives 0 results (known as the "Single-Category Recall" problem).

Pre-Filtering (Standard): The database filters out irrelevant records first using SQL-like metadata boundaries, and then runs the vector similarity search strictly on the matching subset. This guarantees retrieving the correct number of top rules.
# Post-Filtering (Unsafe) Search Globals → Filter workspace (Can yield 0 hits if Top-K matches aren't yours) # Pre-Filtering (Safe - ChromaDB) collection.query( query_embeddings=[q_vector], where={"workspace_id": "ws1"}, # metadata filter n_results=3 )
🔗 4. Hybrid Search: Dense vs. Sparse (RRF)
RAG retrievers often merge two different search methodologies:

Dense Retrieval (Neural): Uses embeddings (like all-MiniLM-L6-v2) to capture conceptual similarity and meaning. It is strong for matching synonyms ("profit" → "revenue minus cost").

Sparse Retrieval (Lexical): Uses word frequency/lexical search (like BM25 or TF-IDF) to search exact keyword tokens. It is strong for matching unique serial IDs, product numbers, or codes.

Reciprocal Rank Fusion (RRF): Merges matches by summing reciprocal ranks: Score = 1/(Rank_dense + 60) + 1/(Rank_sparse + 60) to return a balanced list of results.
[Hybrid Search Score Blending] Vector Similarity: Captures semantic synonyms (70%) Lexical Keyword: Captures exact code/word hits (30%) final_score = (dense * 0.7) + (sparse * 0.3)
🗑️ 5. Context Window Management
LLMs have strict limits on the number of context tokens they can process. If you retrieve too many rules or schema mappings, the prompt overflows, causing either API errors or model "attention dilution" where it ignores instructions.

Mitigation: 1. **Rank & Truncate**: Filter and sort using hybrid scores, only keeping the top K rules. 2. **Tokens Budgeting**: Dynamically count query tokens and slice context arrays before hitting prompt boundaries.
# Dynamic Context Budgeting Total Limit: 4096 tokens User query: 50 tokens SQLite Schemas: 1200 tokens Remaining Budget: 2846 tokens → Fill with top rules until budget is consumed.
🛡️ 6. Confidence Scoring & Clarification Gatekeeping
Before passing the built context to the LLM for SQL generation, the pipeline runs a Confidence Scorer to evaluate query completeness and mapping quality at runtime.

Why it is in Phase 2 Only: Phase 1 (Ingestion) only writes embeddings statically. Phase 2 (Retrieval) dynamically evaluates queries. If the query is ambiguous, missing join definitions, or matching low-similarity rules, the score falls below the threshold.

Clarification Engine: If the confidence score is below 70, the engine prompts the user for specific dimensions or metrics rather than executing a guessed, potentially incorrect SQL query.
# Confidence Threshold Guard If score < 70: halt_execution() trigger_clarification_questions() else: generate_and_execute_sql() # Fallback Scoring Formula Score = MetadataMatch(30) + JoinMatch(20) + VectorSim(35) + IntentParsed(15)
